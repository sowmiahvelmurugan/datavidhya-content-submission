# Data Warehouse vs Data Lake vs Lakehouse

Imagine you're a librarian. When the library first opened, it only stocked hardcover books — everything
was carefully catalogued, spine-up, easy to find. Then the internet happened and suddenly you're also
getting PDFs, audio recordings, YouTube transcripts, and raw sensor data from a smart building.
Your old shelving system doesn't work anymore.

That's roughly the problem data engineering faced between the 1990s and today. The three solutions —
**data warehouses**, **data lakes**, and **lakehouses** — each represent a different answer to the
same question: *how do we store and make sense of all this data?*

By the end of this article, you'll understand what each architecture is, why it was built, where it breaks
down, and — most importantly — how to choose between them. These aren't just theoretical concepts.
This decision shapes how every analytics team at every company operates day to day.

---

## Data Warehouse

A **Data Warehouse** is a centralized repository designed to store **structured data** for analytical
purposes. It primarily holds historical data rather than day-to-day transactional data. Think of it as a
**highly organized spreadsheet** that consolidates data from multiple sources so analysts can query it
reliably and fast.

### How Data Gets In: ETL + Schema-on-Write

Before any data lands in a warehouse, its **schema** — the structure, column names, and data types —
must be defined upfront. Every incoming record is cleaned and reshaped to match that schema before
it's stored. This is called **schema-on-write**, and the process that enforces it is called **ETL**:

1. **Extract** — Collect raw data from source systems (CRMs, databases, APIs)
2. **Transform** — Clean, validate, and reshape data to fit the predefined schema
3. **Load** — Insert the transformed data into the warehouse

Once data is inside, analysts query it with SQL. Warehouses use **columnar storage** — instead of
storing each row together, each column is stored separately. This makes analytical queries dramatically
faster because reading `SUM(sales_amount)` only touches one column, not every field in every row.

```sql title="simple_warehouse_query.sql"
-- Columnar storage means this query only reads 2 columns
-- even if the table has 50+ columns -- that's what makes it fast
SELECT
  product_name,
  SUM(sales_amount) AS total_sales
FROM sales
GROUP BY product_name
ORDER BY total_sales DESC;
```

<Callout type="key-insight" title="Key Insight">
A data warehouse enforces structure **at write time** — before data even lands. This is why queries are
fast and data quality is guaranteed. But it also means you must know what questions you want to ask
*before* you store the data. That assumption breaks down the moment your data stops being purely
structured.
</Callout>

**Best for:** BI dashboards, batch reporting, historical trend analysis, executive metrics.

**Popular examples:** Amazon Redshift, Google BigQuery, Snowflake, Azure Synapse.

**Limitations:**
- Schema must be defined upfront — adding a new column means altering the schema and re-running ETL
- Cannot store unstructured data: images, videos, raw logs, or JSON blobs
- Expensive at very high storage volumes (compute and storage are often bundled)
- Not designed for ML model training or data science workflows

## Data Lake

A **Data Lake** is a centralized storage system that holds **large volumes of raw data in its native
format** — structured (CSV, Parquet), semi-structured (JSON, XML), or completely unstructured
(images, audio, server logs, clickstreams). No predefined schema required.

### How Data Gets In: ELT + Schema-on-Read

Instead of enforcing structure before storage, data lakes apply structure **at read time** — called
**schema-on-read**. The process is **ELT** (Extract, Load, Transform), which is the reverse of ETL:

1. **Extract** — Pull data from source systems
2. **Load** — Dump raw data directly into object storage (S3, ADLS, GCS) immediately
3. **Transform** — Process and structure data only when a specific use case needs it

Because data lands in raw form, you keep everything — even fields you didn't know you'd need later.
A data scientist building a fraud model six months from now can access the same raw logs your
operations team ingested today.

```sql title="schema_on_read_example.sql"
-- raw_timestamp is stored as plain text in the lake
-- We apply structure only when reading -- this is schema-on-read
SELECT
  user_id,
  page_url,
  TO_TIMESTAMP(raw_timestamp, 'dd/MM/yyyy HH:mm:ss') AS visit_time
FROM raw_clickstream_logs
WHERE event_type = 'page_view';
```

<Callout type="info" title="Context">
**ETL vs ELT — why does the order matter?**
ETL (warehouse-era) enforces schema before storing, which means you may discard data you didn't
anticipate needing. ELT (lake-era) stores everything raw first, so you can always reprocess historical
data differently. This shift happened because cloud storage became nearly free and cloud compute
became powerful enough to transform data in-place — making a separate transformation step before
loading unnecessary.
</Callout>

  src="./images/data-lake-architecture.png"

  caption="A data lake stores everything raw — structure is applied at read time, not write time"
/>

**Best for:** ML training data, raw log storage, data science exploration, storing data you don't know
how to use yet.

**Popular examples:** Amazon S3 + Apache Spark, Azure Data Lake Storage Gen2, Google Cloud
Storage, Apache Hadoop/HDFS (legacy on-premise).

**Limitations:**
- No ACID transactions — two writers can corrupt the same data; updating or deleting individual records
  is painful
- Query performance is slow without careful partitioning and file format optimization
- Not analyst-friendly — requires Spark or engines like Athena/Presto for basic queries
- Without governance, data lakes become **data swamps**

<Callout type="warning" title="Common Mistake">
Treating a data lake as "just a cheaper data warehouse" is a classic interview mistake. A data lake is a
**fundamentally different architecture** — not a warehouse with lower storage costs. If you try to use a
data lake as a warehouse without a table format layer on top, you get slow queries, no ACID guarantees,
and eventual data quality disasters. The interviewer wants to hear you explain *why* they're different, not
just that they cost different amounts.
</Callout>

## Data Lakehouse

A **Data Lakehouse** is the newest architecture, pioneered by Databricks with Delta Lake around 2020.
The idea is elegant: keep the cheap, flexible, open storage of a data lake, but add a **table format
layer** on top that gives you warehouse-grade reliability and SQL performance.

<Callout type="key-insight" title="Key Insight">
A lakehouse isn't a new type of storage — it's a **software layer** on top of object storage. Tools like
**Delta Lake**, **Apache Iceberg**, and **Apache Hudi** add ACID transactions, schema enforcement,
versioning, and query metadata to plain Parquet files sitting on S3. You get the economics of a data
lake with the reliability and performance approaching a warehouse — on a single copy of data.
</Callout>

Think of it this way: S3 is a dumb file system. Parquet is a great column format. But neither one knows
what a "table" is, or how to handle two writers updating the same file concurrently. The table format
layer adds a **transaction log** on top of those Parquet files — so now your lake has tables, ACID
guarantees, and time travel (the ability to query data as it existed at any point in the past).

### How It Works

The lakehouse lets you use plain SQL for operations that were previously impossible in a data lake —
upserts, deletes, and querying historical snapshots. Here's what that looks like in practice.

**1. Create a Delta / Iceberg table on top of object storage**

```sql title="create_lakehouse_table.sql"
-- Create a Delta table stored in S3 -- this is just Parquet + a transaction log
-- The table format layer (Delta/Iceberg) manages all the metadata on top
CREATE TABLE customers (
  customer_id   BIGINT,
  full_name     STRING,
  email         STRING,
  country       STRING,
  updated_at    TIMESTAMP
)
USING DELTA
LOCATION 's3://lake/gold/customers/';
```

**2. MERGE (Upsert) — the operation a plain data lake cannot do**

```sql title="lakehouse_merge_upsert.sql"
-- MERGE is the killer feature of lakehouses
-- A plain S3 data lake has no concept of "update this specific row"
-- Delta/Iceberg adds ACID guarantees -- if two jobs run simultaneously,
-- one waits. Neither corrupts the table.

MERGE INTO customers AS target
USING customer_updates AS source
  ON target.customer_id = source.customer_id

-- If the customer already exists, update their record
WHEN MATCHED THEN
  UPDATE SET
    target.full_name   = source.full_name,
    target.email       = source.email,
    target.country     = source.country,
    target.updated_at  = source.updated_at

-- If they are new, insert them
WHEN NOT MATCHED THEN
  INSERT (customer_id, full_name, email, country, updated_at)
  VALUES (source.customer_id, source.full_name, source.email,
          source.country, source.updated_at);
```

**3. Time Travel — query data as it existed at any past point**

```sql title="lakehouse_time_travel.sql"
-- Query the customers table as it looked on March 4th
-- Every change is tracked in the transaction log, so this just works
-- Useful for: debugging pipeline issues, audits, and GDPR requests

-- Delta Lake syntax (Databricks / Spark SQL)
SELECT * FROM customers
TIMESTAMP AS OF '2026-03-04';

-- Apache Iceberg syntax (Trino, Spark, Flink, DuckDB)
SELECT * FROM customers
FOR SYSTEM_TIME AS OF TIMESTAMP '2026-03-04 00:00:00';

-- Roll back the entire table to a previous version (the "undo" button)
RESTORE TABLE customers TO TIMESTAMP AS OF '2026-03-04';
```

**4. GDPR-compliant row-level delete — impossible in a plain data lake**

```sql title="lakehouse_gdpr_delete.sql"
-- A user requests to be forgotten under GDPR
-- In a plain data lake you would have to rewrite entire Parquet files
-- In a lakehouse, this is a single SQL statement
DELETE FROM customers
WHERE customer_id = 98765;
```

### The Three Table Formats

These power every major lakehouse deployment. Knowing the differences matters in interviews:

- **Delta Lake** — Created by Databricks. Best-in-class for Spark workloads. Most widely adopted.
- **Apache Iceberg** — Built at Netflix. Cloud-agnostic, works with any query engine (Spark, Flink,
  Trino, DuckDB). Preferred when you want zero vendor lock-in.
- **Apache Hudi** — Invented at Uber for high-frequency row-level upserts and deletes at massive scale.

  caption="The lakehouse unifies analytics and ML on a single copy of data — no ETL copying between a lake and a separate warehouse"
/>

**Best for:** Teams that need both SQL analytics and ML workloads on the same data, GDPR compliance
(row-level deletes), real-time + batch on one platform.

**Popular examples:** Databricks + Delta Lake, Apache Iceberg on S3, AWS Lake Formation + Iceberg.

**Limitations:**
- More complex to operate than a managed warehouse — requires Spark, cluster management, and tuning
- Pure SQL query performance still lags behind purpose-built warehouses like Snowflake for heavy BI
- Tooling is maturing fast but is younger — fewer out-of-the-box BI connector integrations

## Data Warehouse vs Data Lake vs Data Lakehouse

| | Data Warehouse | Data Lake | Data Lakehouse |
|---|---|---|---|
| **Data types** | Structured only | Any (structured, semi, unstructured) | Any |
| **Schema** | Enforced at write (ETL) | Applied at read (ELT) | Enforced at write (flexible) |
| **Query performance** | Excellent | Poor to moderate | Good (improving fast) |
| **ACID transactions** | Yes | No | Yes (Delta/Iceberg/Hudi) |
| **Storage cost** | High | Low | Low |
| **ML workloads** | Poor fit | Good fit | Excellent fit |
| **Update/delete rows** | Supported | Not natively | Yes (MERGE/UPSERT) |
| **Time travel** | Limited | No | Yes |
| **Best for** | BI, dashboards, reporting | Raw storage, ML training | Unified analytics + ML |
| **Examples** | Snowflake, BigQuery, Redshift | S3+Spark, ADLS, GCS | Databricks, Iceberg on S3 |

## Decision Framework: Which Should You Use?

```
Start here: What does your team primarily need to do?
│
├── Pure SQL analytics, dashboards, and BI reporting?
│   └── → Data Warehouse  (Snowflake, BigQuery, Redshift)
│
├── Store massive raw data cheaply — ML training, logs, no immediate query needs?
│   └── → Data Lake on object storage  (S3/GCS + Spark)
│
├── Both analytics AND ML on the same data, okay with more complexity?
│   └── → Lakehouse  (Databricks + Delta Lake, or Iceberg on S3)
│
└── Starting from scratch, small team, mostly structured data?
    └── → Start with a cloud warehouse. You can add lake/lakehouse later.
```

<Callout type="tip" title="Pro Tip">
In practice, most mature data platforms are **hybrid**: raw data lands in a lake (Bronze layer), curated
tables live in a lakehouse (Silver layer), and fast aggregates are served from a warehouse or Gold
lakehouse layer. This pattern — called the **Medallion Architecture** (Bronze → Silver → Gold) — is
used at Databricks, Uber, LinkedIn, and most companies building modern data platforms. If an
interviewer asks "how would you design a data platform from scratch?", describing this pattern with
justification for each layer will set you apart.
</Callout>

## Real-World Examples

### Uber: Inventing Apache Hudi

Uber's early analytics infrastructure relied on Hadoop-based data lakes. The problem: drivers complete
millions of trips per day, and updating earnings records, trip statuses, and surge pricing meant
constantly rewriting massive files. Classic data lakes have no efficient "update a record" operation —
you'd rewrite the entire partition every time.

Uber built **Apache Hudi** (Hadoop Upserts Deletes and Incrementals) to solve exactly this. Hudi added
record-level upserts and deletes to their data lake — essentially building one of the first lakehouse table
formats. Today Hudi processes exabytes of data at Uber.

The lesson: when you have high-frequency updates to large datasets, a plain data lake breaks down.
You need the lakehouse layer.

### Netflix: The Multi-Layer Approach

Netflix uses a layered architecture that perfectly illustrates the hybrid pattern. Raw event data —
viewing sessions, clicks, encoder outputs — lands in S3 as a data lake. It's processed with Spark and
stored in **Apache Iceberg** tables, which serve their data science and ML teams building
recommendation models. For executive dashboards and finance reporting, aggregated data is loaded
into **Snowflake** for fast, reliable SQL queries.

Netflix doesn't force everything into one system. They use each architecture where it's strongest. That's
the real-world answer: not "which is best" but "which is best *for this specific use case*."

## Common Misconceptions

### 1. "A data lake is just a cheaper data warehouse"

This will cost you in interviews. A data lake is a fundamentally different architecture — not a warehouse
with lower storage costs. A warehouse enforces structure and guarantees quality at write time. A data
lake stores raw, unmodified data and applies structure at read time. They serve different purposes. If
you treat a data lake as a cheap warehouse without adding a table format layer, you'll end up with slow
queries, no ACID guarantees, and data quality problems within months.

### 2. "Lakehouses make data warehouses obsolete"

Not so fast. Cloud data warehouses (Snowflake, BigQuery) are still faster for pure SQL analytics,
simpler to operate, and more mature in governance tooling. For teams that primarily do SQL-based BI
without ML workloads, a warehouse is often still the right choice — it's easier and faster to query. The
lakehouse is compelling when you *need* to unify analytics and ML on one platform, but that's not every
company, and the operational complexity is real.

### 3. "ETL and ELT are just the same thing in a different order"

The order is everything. ETL transforms before storing — meaning you may discard raw data you didn't
anticipate needing. ELT stores everything raw first, so you can always reprocess historical data for new
use cases. This is why data lakes and lakehouses fundamentally expanded what's possible in data
engineering: raw data preservation unlocks future ML workloads that no one planned for at ingestion time.

## Interview Tip

<Callout type="interview" title="Interview Relevance">
This topic is a staple in data engineering interviews at all levels. Here's how it actually comes up:

**Conceptual questions (entry/mid-level):** "What's the difference between a data lake and a data
warehouse?" — Expected answer: schema-on-write vs schema-on-read, structured vs any format, ACID
vs no ACID. Don't just say "one is cheaper."

**Follow-up trap:** "Why not just use a data lake for everything?" — The interviewer wants to hear
about data swamps, the lack of ACID transactions, query performance limitations, and governance
challenges. Saying "it's slower" alone won't cut it.

**System design round (mid/senior):** "Design a data platform for an e-commerce company with 10M
daily transactions." — You're expected to describe an ingestion layer, a storage layer (lake, warehouse,
or lakehouse), and a serving layer. Choosing the right architecture *and justifying it* is the point.

**Senior-level framing:** At senior and staff levels at companies like Meta, Airbnb, and Stripe,
interviewers will ask you to weigh trade-offs based on team size, data volume, query patterns, and ML
needs — not just recite definitions. Knowing the Medallion Architecture (Bronze/Silver/Gold) and being
able to explain *why* Uber built Hudi will set you apart from candidates who only know the theory.
</Callout>

---

<KeyTakeaways>
- **Data Warehouses** enforce schema at write time (ETL), store structured data only, and are optimized for fast SQL analytics and BI — the right tool for dashboards and reporting, but expensive at scale and inflexible for ML.
- **Data Lakes** store raw data of any type cheaply on object storage, applying structure at read time (ELT) — great for ML training data and raw ingestion, but prone to becoming unqueryable data swamps without governance.
- **Lakehouses** add ACID transactions, schema enforcement, and time travel (via Delta Lake, Iceberg, or Hudi) on top of data lake storage — combining warehouse reliability with lake flexibility and economics.
- **In practice, most mature platforms are hybrid**: raw data lands in a lake, curated tables live in a lakehouse layer, and fast aggregates are served from a warehouse. The Medallion Architecture (Bronze → Silver → Gold) is the industry-standard pattern for organizing this.
- **The key interview insight**: choose based on your team's primary workloads — pure SQL analytics favors a warehouse; ML + analytics together favors a lakehouse; raw storage at massive scale favors a lake. "It depends" is the right answer when you can back it up.
</KeyTakeaways>
